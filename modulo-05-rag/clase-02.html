Clase 02: Context stuffing en serio — ventanas largas y PDFs de 200 páginas

📅 Semana 6 — Martes | ⏱️ 50 minutos | 🧭 La opción que muchos ignoran

🧭 Por qué esta clase importa

En la Clase 01 viste que tienes 3 arquitecturas. Hoy la llevamos a fondo: context stuffing no es "el truco de principiante antes de montar RAG". En 2026, con ventanas de 200K–1M tokens y prompt caching, stuffing gana a RAG clásico en la mayoría de casos prácticos de menos de 500 páginas.

Al acabar la clase sabrás: cuántos tokens pesa realmente un PDF largo, cómo meterlo entero en Claude Sonnet 4.6, cómo activar prompt caching para que las siguientes preguntas cuesten 10× menos, y cómo detectar cuándo el modelo empieza a sufrir "lost in the middle".

📚 Resumen de Conceptos

¿Qué significa "200K tokens" en la vida real?

Los tokens no son palabras. En español, la regla que casi nunca falla:

  • 1 token ≈ 4 caracteres (con espacios incluidos)
  • 1 palabra española ≈ 1,3–1,5 tokens
  • 1 página de PDF (texto normal) ≈ 500–700 tokens

Con esa regla, la ventana de 200 000 tokens de Claude Sonnet 4.6 te da ~280–400 páginas de PDF. La versión 1M (beta/long-context) llega a ~1 500 páginas: un libro técnico completo más documentación.

Intuición rápida: si puedes leer tu doc en una tarde, cabe en el prompt. Si llenaría una estantería, probablemente no.

Los 3 problemas reales del stuffing (y cómo se resuelven en 2026)

Problema 1 — Coste: pagas todos los tokens en cada llamada

Claude Sonnet 4.6 (abril 2026) cobra ~3 $/M tokens de input. Un PDF de 200 páginas ≈ 130 K tokens. Cada pregunta cuesta ~0,39 $ solo en input. Diez preguntas al día = 4 $/día = 120 $/mes solo por hacer Q&A sobre un documento.

Solución 2026 → Prompt caching. Anthropic te deja marcar bloques de prompt como cacheados. El primer envío cuesta 1,25× (sobrecoste de escritura); los siguientes, durante 5 min, cuestan 0,1×. Traducido: 10× más barato a partir de la segunda pregunta sobre el mismo documento.

Problema 2 — "Lost in the middle"

Demostrado por Liu et al. (2023) y replicado infinitas veces: cuando el modelo recibe un contexto muy largo, tiende a recordar mejor el principio y el final que la zona central. Es una curva en forma de U.

En 2026 Claude Sonnet 4.6 lo mitiga bastante — en la evaluación "needle in a haystack" alcanza > 99 % hasta los 200 K tokens — pero sigue existiendo cuando metes docs ruidosos, estructura pobre o múltiples documentos juntos.

Mitigaciones prácticas:

  • Pon la pregunta al final del prompt, nunca al principio.
  • Añade cabeceras claras al documento (## Sección 3 — Devoluciones) para que el modelo pueda "saltar" mentalmente.
  • Si metes varios documentos, sepáralos con marcadores: <doc id="manual-v2">...</doc>.
  • Si cuelga el resultado, prueba a partir el doc por la mitad y hacer dos llamadas — es el diagnóstico más barato antes de montar RAG.

Problema 3 — Latencia

Meter 200 K tokens no es gratis en tiempo. El time-to-first-token con prompt cacheado baja de ~8–10 s a ~1–2 s. Sin caché, prepara a Juan para esperar unos segundos; con caché, la experiencia es casi interactiva.

Cuándo stuffing gana a RAG clásico (la sorpresa del módulo)

Si tu corpus cumple las tres condiciones de abajo, stuffing con prompt caching es más simple, más barato operativamente (sin infra) y a menudo igual o mejor de preciso que un RAG clásico:

  1. Volumen < 200 K tokens (~300 páginas)
  2. Cambios poco frecuentes (el caché se aprovecha)
  3. Preguntas que requieren mirar varias partes del doc a la vez (cross-sección)

Esta es la trampa en la que caen los juniors: montar RAG por defecto cuando el doc cabe en el contexto. En 2026 eso cuesta tiempo, complejidad y suele dar peores respuestas.

Comparativa operativa

Aspecto Stuffing sin caché Stuffing con caché RAG clásico
Coste 1ª query (200 pág) ~0,39 $ ~0,49 $ (escribe caché) ~0,003 $ (solo chunks relevantes)
Coste 10ª query ~0,39 $ ~0,04 $ 🟢 ~0,003 $
Infra Nada 🟢 Nada 🟢 Vector DB + embeddings

Lo que vale en tu CV

Saber cargar un PDF, medir tokens, activar caché y razonar "aquí stuffing es suficiente" te diferencia del 90 % de juniors que saltan a RAG por reflejo. Este es un skill de arquitectura, no de copy-paste.

⚙️ Práctica Guiada: un PDF de 200 páginas, con caché, y Q&A real

Vamos a coger un PDF grande (un manual técnico o un paper largo), meterlo entero en un prompt de Claude y hacerle preguntas. Medimos tokens, medimos coste, activamos prompt caching y comprobamos la diferencia.

Paso 1: Instalar dependencias

pip install anthropic python-dotenv pypdf

Sigue sin haber LangChain. pypdf es el parser estándar y es una librería de 1 archivo, no un framework.

Paso 2: Conseguir un PDF grande de prueba

Si no tienes uno a mano, descárgate uno real (publicaciones abiertas):

  • Un paper largo de arXiv (muchos tienen 50-80 páginas).
  • El informe anual de Amazon Science u otra publicación corporativa.
  • Un libro en dominio público (Project Gutenberg → PDF).

Guárdalo como manual.pdf en la raíz del proyecto.

Paso 3: Cargar el PDF y contar tokens

import os
from pathlib import Path
from pypdf import PdfReader
from anthropic import Anthropic
from dotenv import load_dotenv

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

# 1. Cargar el PDF a texto plano
reader = PdfReader("manual.pdf")
texto = "\n\n".join(
    f"## Página {i+1}\n{page.extract_text() or ''}"
    for i, page in enumerate(reader.pages)
)

print(f"Páginas extraídas: {len(reader.pages)}")
print(f"Caracteres totales: {len(texto):,}")
print(f"Tokens estimados (chars/4): {len(texto)//4:,}")

# 2. Contar tokens exactos con el endpoint de Anthropic
respuesta_count = client.messages.count_tokens(
    model="claude-sonnet-4-6",
    messages=[{"role": "user", "content": texto[:100_000]}]
)
print(f"Tokens reales (primeros 100K chars): {respuesta_count.input_tokens:,}")

Fíjate en la diferencia entre estimación (chars/4) y tokens reales. Para español suelen coincidir ± 10 %. El contador de Anthropic es la referencia.

Paso 4: Primera pregunta — stuffing sin caché

pregunta = "¿Cuáles son las 3 ideas principales del documento? Cita las páginas."

prompt = f"""Usa SOLO la información del documento para responder. Si no está, di "no está en el documento".

DOCUMENTO:
{texto}

PREGUNTA: {pregunta}"""

import time
t0 = time.time()
resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": prompt}]
)
latencia = time.time() - t0

print(resp.content[0].text)
print(f"\n⏱️  Latencia: {latencia:.1f}s")
print(f"📊 Tokens in: {resp.usage.input_tokens:,}  out: {resp.usage.output_tokens:,}")

coste = (resp.usage.input_tokens * 3 + resp.usage.output_tokens * 15) / 1_000_000
print(f"💸 Coste: ${coste:.4f}")

Paso 5: Activar prompt caching

Este es el cambio. El doc va en system con cache_control. Primera llamada: escribe el caché (1,25× el coste base). Siguientes llamadas en los próximos 5 min: 10× más baratas. 

# Nota: prompt caching se activa en el bloque system con cache_control
system_con_cache = [
    {
        "type": "text",
        "text": f"Eres un asistente experto en este documento. Usa SOLO el documento para responder.\n\nDOCUMENTO:\n{texto}",
        "cache_control": {"type": "ephemeral"}
    }
]

def preguntar_con_cache(pregunta: str):
    t0 = time.time()
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system_con_cache,
        messages=[{"role": "user", "content": pregunta}]
    )
    latencia = time.time() - t0
    u = resp.usage
    print(f"\n❓ {pregunta}")
    print(resp.content[0].text[:300] + "...")
    print(f"⏱️  {latencia:.1f}s  |  in: {u.input_tokens}  out: {u.output_tokens}"
          f"  |  cache_read: {getattr(u, 'cache_read_input_tokens', 0)}"
          f"  |  cache_write: {getattr(u, 'cache_creation_input_tokens', 0)}")
    return resp

# 1ª llamada — ESCRIBE caché (pagas 1,25× en input_tokens de la parte cacheada)
preguntar_con_cache("Resume el documento en 5 puntos.")

# 2ª, 3ª, 4ª llamadas — LEEN caché (pagas 0,1× en input_tokens de la parte cacheada)
preguntar_con_cache("¿Menciona el documento alguna limitación o gap?")
preguntar_con_cache("Dame 3 citas textuales que apoyen la idea principal.")
💡 Observa los campos cache_read_input_tokens y cache_creation_input_tokens en el usage. La primera llamada tiene cache_creation alto y cache_read a cero. A partir de la segunda, se invierte. Esto es lo que quieres ver en logs de producción.

Paso 6: Calcula el ahorro real

# Precios Claude Sonnet 4.6 (abril 2026) — verifica en docs:
# input normal:        $3 / M tokens
# cache write (5 min): $3.75 / M tokens (1,25×)
# cache read:          $0.30 / M tokens (0,1×)
# output:              $15 / M tokens

def coste(u):
    normal = (u.input_tokens or 0) * 3
    write  = (getattr(u, "cache_creation_input_tokens", 0) or 0) * 3.75
    read   = (getattr(u, "cache_read_input_tokens", 0) or 0) * 0.30
    out    = (u.output_tokens or 0) * 15
    return (normal + write + read + out) / 1_000_000

# Compara tus 3 llamadas cacheadas con una llamada sin caché
# Deberías ver que la llamada 3 cuesta ~10× menos que la 1
⚠️ Advertencia importante: el caché expira a los 5 minutos de inactividad. Si dejas una pausa larga entre preguntas, pagas la escritura otra vez. Para producción hay caché extendido (1 h) con coste distinto — míralo en la doc oficial.

🎯 Mini-Reto del Día: doc entero vs doc partido

Desafío: demuestra empíricamente si "lost in the middle" te afecta con tu documento.

Requisitos:

  1. Coge el mismo PDF de la práctica (idealmente > 80 páginas).
  2. Diseña una pregunta cuya respuesta esté en la página central del doc (mira tú el PDF y escoge algo concreto, que sepas que está ahí).
  3. Haz la llamada A: stuff el doc entero y pregunta.
  4. Haz la llamada B: parte el doc en 2 mitades; haz dos llamadas (una a cada mitad) y pregunta a ambas. Quédate con la respuesta no vacía.
  5. Compara: ¿ha encontrado la respuesta en A? ¿En B? ¿Cuál ha sido más precisa, más corta, más citable?

Código de inicio:

import os, time
from pathlib import Path
from pypdf import PdfReader
from anthropic import Anthropic
from dotenv import load_dotenv

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

reader = PdfReader("manual.pdf")
paginas = [p.extract_text() or "" for p in reader.pages]
n = len(paginas)
mid = n // 2

doc_entero  = "\n\n".join(f"## p.{i+1}\n{t}" for i, t in enumerate(paginas))
doc_primera = "\n\n".join(f"## p.{i+1}\n{t}" for i, t in enumerate(paginas[:mid]))
doc_segunda = "\n\n".join(f"## p.{i+1}\n{t}" for i, t in enumerate(paginas[mid:], start=mid))

PREGUNTA = "¿Qué dice el documento sobre [ESCRIBE AQUÍ algo que esté en la zona central]?"

def ask(doc, pregunta):
    prompt = f"Usa SOLO este documento. Si no está, di 'no está'.\n\n{doc}\n\nPREGUNTA: {pregunta}"
    r = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=800,
        messages=[{"role": "user", "content": prompt}]
    )
    return r.content[0].text, r.usage

print("=== A: doc entero ===")
txt_a, u_a = ask(doc_entero, PREGUNTA)
print(txt_a)
print(f"tokens in/out: {u_a.input_tokens}/{u_a.output_tokens}")

print("\n=== B: doc partido (mitad 1) ===")
txt_b1, u_b1 = ask(doc_primera, PREGUNTA)
print(txt_b1)

print("\n=== B: doc partido (mitad 2) ===")
txt_b2, u_b2 = ask(doc_segunda, PREGUNTA)
print(txt_b2)

Bonus: hazlo con 3 preguntas distintas (una de la página 10, una de la página central, una de la última). Apunta cuál estrategia gana en cada caso. Eso es un mini-eval.
