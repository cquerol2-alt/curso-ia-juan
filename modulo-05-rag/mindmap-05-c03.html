Mind Map — Clase 03: agentic search

El modelo decide

Tool use nativo de Anthropic: bucle ReAct, sin LangChain

🤖
¿Qué es tool use?
+
Contrato: el modelo pide llamadas a funciones; tu código las ejecuta
Le pasas tools=[...] a messages.create; él devuelve bloques tool_use
Señal clave: resp.stop_reason == "tool_use" → hay que ejecutar y devolver tool_result
Así funcionan Claude Code, Cursor, Perplexity — no tienen "el doc entero" cargado, lo buscan
Nombre técnico: function calling (OpenAI) = tool use (Anthropic). Mismo concepto.
🔄
El bucle ReAct (Reason + Act)
+
1. Llama a la API con tools y messages
2. Si stop_reason != "tool_use" → respuesta final, break
3. Añade el turno del assistant a messages (bloque tool_use)
4. Ejecuta cada tool; añade {role:user, content:[{type:"tool_result", tool_use_id:..., content:...}]}
5. Vuelve a 1. Siempre con max_turnos (8–12) para evitar loops infinitos
Mental model: tu programa es un loop; el "cerebro" del agente es el modelo dentro del loop
🧬
Anatomía de una tool
+
name: identificador snake_case único (ej: search_docs)
description: el 80% del trabajo. Dile qué hace, qué devuelve y cuándo usarla
input_schema: JSON Schema con type, properties, required
Regla: si dudas si el modelo la llamará cuando toca → tu description está floja
Trampa: tools con nombres idénticos o descriptions solapadas → el modelo se confunde. Sé específico
✅
Ventajas vs stuffing
+
Escala infinita: el corpus puede tener 10 GB; solo se leen los fragmentos que el agente decide abrir
Datos vivos: tools pueden leer DB, API, web en tiempo real. Stuffing es estático
Pago proporcional: una query barata cuesta poco, una compleja cuesta más. Más honesto que "paga el doc entero"
Multi-paso: el agente puede encadenar "busca X → lee archivo → extrae dato" sin que lo orquestes tú
Debug observable: ves los tool_use turno a turno → razonamiento explícito
🎯
Cuándo agentic gana (y cuándo no)
+
Gana cuando: corpus no cabe en 200K tokens, datos cambian, info no predecible, workflows multi-paso
Pierde cuando: un doc cabe + se consulta muchas veces → stuffing + caché es más simple y más rápido
Pierde cuando: latencia crítica (sub-segundo) → cada turno suma 1–3 s
Riesgo real: tool mal descrita → el agente la llama cuando no toca, o no la llama cuando sí. Iterar descriptions es parte del trabajo
Seguridad: tools con efectos (escribir archivos, ejecutar SQL) requieren validación estricta de inputs. El modelo NO es tu firewall
