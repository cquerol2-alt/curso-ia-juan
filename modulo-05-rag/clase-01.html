Clase 01: El problema — Cómo conectas una IA con tus datos

📅 Semana 6 — Lunes | ⏱️ 45 minutos | 🧭 Clase bisagra del módulo

🧭 Por qué esta clase es distinta

Hasta 2024, si alguien te preguntaba "¿cómo hago que un LLM responda sobre mis documentos?", la respuesta automática era "móntale un RAG con LangChain". En 2026 eso ya no es verdad. Tienes 3 formas de darle a una IA información que no está en su entrenamiento — y elegir bien es más importante que saber programar cualquiera de las tres.

En esta clase no vas a tocar embeddings ni vector stores todavía. Vas a aprender a pensar el problema antes de elegir herramienta. Esto es lo que en 2026 se llama context engineering, y es la habilidad que separa a un junior que "sabe LangChain" de un dev que entiende el stack.

📚 Resumen de Conceptos

El problema real: el "apagón de conocimiento"

Un LLM como Claude o GPT fue entrenado con datos hasta una fecha concreta (su knowledge cutoff). Pregúntale por:

  • Tus apuntes de la U-TAD del último trimestre
  • El manual interno de una empresa
  • Las últimas 3 entregas de tu equipo en Linear
  • Un paper publicado la semana pasada

…y lo va a inventar con confianza absoluta. A esto se le llama alucinación. El modelo no sabe que no sabe.

Context engineering es la disciplina de meterle al modelo la información correcta en el momento correcto para que deje de inventar. No es prompt engineering (eso es "cómo le hablo"). Es "qué datos le doy".

Las 3 arquitecturas — de menor a mayor complejidad

1️⃣ Context stuffing (meterlo todo en el prompt)

Los modelos de 2026 tienen ventanas de contexto gigantes: Claude Sonnet 4.6 maneja 200 000 tokens (≈ 150 000 palabras, un libro entero) y versiones "long" llegan a 1 millón. Si tu conocimiento cabe ahí dentro, meterlo entero en el prompt es la solución más simple.

  • Cuándo usar: documentos < 200K tokens, consultas puntuales, prototipado rápido
  • Ventajas: cero infraestructura, latencia baja, el modelo ve todo el contexto
  • Desventajas: caro por llamada (pagas todos los tokens siempre), problema de "lost in the middle" (el modelo ignora información en medio de contextos muy largos)

2️⃣ Agentic search (que el modelo decida qué buscar)

En vez de darle todo, le das al modelo herramientas (tools): search_docs(), read_file(), web_search(). El modelo lee la pregunta, decide qué necesita buscar, ejecuta la herramienta, lee el resultado y responde. Esto es lo que hacen Claude Code, Cursor o Perplexity por debajo.

  • Cuándo usar: corpus grandes dinámicos, preguntas que requieren buscar en varias fuentes, workflows donde la información no es predecible
  • Ventajas: el modelo escala a cualquier tamaño de datos, solo paga los tokens que necesita, se adapta a la pregunta
  • Desventajas: más latencia (múltiples turnos), requiere construir tools fiables, más complejidad de debug

3️⃣ RAG clásico (embeddings + vector store + retrieval)

Chunks + embeddings + vector DB + retrieve top-K + prompt aumentado + respuesta. Lo de siempre. Lo importante en 2026 es entender que es una opción entre tres, no la opción por defecto.

  • Cuándo usar: corpus muy grandes (millones de docs), latencia crítica, coste por consulta bajo, búsqueda semántica donde las tools no bastan
  • Ventajas: escalable a millones de documentos, coste por query bajo y predecible, búsqueda semántica potente
  • Desventajas: pipeline complejo, hay que mantener el índice al día, calidad depende de chunking/reranking

La jerarquía de decisión

Regla práctica de 2026: empieza por stuffing si cabe → si no cabe o necesitas frescura, pasa a agentic search → solo si el volumen o la latencia lo exigen, móntate un RAG clásico.

Comparativa rápida

Criterio Context stuffing Agentic search RAG clásico
Tamaño del corpus < 200K tokens Cualquiera Millones de docs
Complejidad Baja 🟢 Media 🟡 Alta 🔴

Qué cambia en tu CV

En 2024 los recruiters preguntaban "¿sabes LangChain?". En 2026 preguntan "¿cuándo usarías RAG y cuándo no?". Saber las 3 opciones y justificar elecciones vale más que pegar pip install langchain.

⚙️ Práctica Guiada: tu primera llamada con contexto

Vamos a hacer el experimento más simple de context engineering: le pedimos al modelo algo que no sabe, y luego se lo damos en el prompt. Dos llamadas, mismo modelo, respuestas radicalmente distintas.

Paso 1: Instalar el SDK de Anthropic

pip install anthropic python-dotenv

Nota: no instalamos LangChain. En 2026, el SDK oficial de Anthropic (y el de OpenAI) cubren todo lo que necesitas sin añadir abstracciones extra.

Paso 2: Configura tu API key

Crea un archivo .env en la raíz del proyecto:

ANTHROPIC_API_KEY=sk-ant-api03-xxxxxxxxxxxxxxxxx

Puedes conseguir una clave en console.anthropic.com (crea cuenta gratis, tienes ~5 $ de crédito inicial — sobra para todo el módulo).

Paso 3: Pregunta SIN contexto (el modelo alucina)

import os
from anthropic import Anthropic
from dotenv import load_dotenv

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

# Pregunta sobre algo que el modelo NO sabe: tu info privada
pregunta = "¿Cuál es la política de devoluciones de la tienda de Juan?"

respuesta = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    messages=[{"role": "user", "content": pregunta}]
)

print("SIN CONTEXTO:")
print(respuesta.content[0].text)

Vas a ver una de estas dos cosas: (a) el modelo dice "no tengo información sobre la tienda de Juan" (bien), o (b) se lo inventa con seguridad (mal, alucinación).

Paso 4: Misma pregunta CON contexto (context stuffing)

# El "documento privado" que queremos que el modelo use
documento = """
POLÍTICA DE DEVOLUCIONES — Tienda de Juan
=========================================
1. Plazo: 30 días desde la fecha de compra.
2. Condiciones: producto sin usar, con etiqueta original.
3. Reembolso: en el mismo método de pago en 5-7 días hábiles.
4. Envío de devolución: gratis si el error es nuestro, a cargo del cliente si el cliente cambia de opinión.
5. Excepciones: productos personalizados no admiten devolución.
"""

prompt_aumentado = f"""Usa SOLO la información del siguiente documento para responder. Si el documento no tiene la respuesta, di "no está en el documento".

DOCUMENTO:
{documento}

PREGUNTA: {pregunta}"""

respuesta = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    messages=[{"role": "user", "content": prompt_aumentado}]
)

print("\nCON CONTEXTO (stuffing):")
print(respuesta.content[0].text)

Esto es context stuffing en su versión más simple. Acabas de hacer lo que el 80 % de los tutoriales te venden como "RAG", pero sin vector store, sin LangChain y sin embeddings. Para 5 párrafos, el 99 % de las veces no necesitas nada más.

Paso 5: Mide el coste real

# El SDK te devuelve cuántos tokens has usado
print(f"Tokens de entrada: {respuesta.usage.input_tokens}")
print(f"Tokens de salida: {respuesta.usage.output_tokens}")

# Precio aproximado Claude Sonnet 4.6 (abril 2026):
# Input:  $3 / millón de tokens
# Output: $15 / millón de tokens
coste = (respuesta.usage.input_tokens * 3 + respuesta.usage.output_tokens * 15) / 1_000_000
print(f"Coste de esta llamada: ${coste:.6f}")

Guarda este número mentalmente. En la Boss Battle del módulo vas a comparar el coste de las 3 arquitecturas sobre los mismos datos — la diferencia te va a sorprender.

💡 Tip: En 2026, el modelo por defecto no es "el más barato", es "el que acierta sin retries". Un retry cuesta más que usar Sonnet desde el principio.

🎯 Mini-Reto del Día

Desafío: Convierte un apunte tuyo de la U-TAD en un "mini asistente" que solo responda sobre ese documento.

Requisitos:

  1. Coge un apunte en texto plano (o conviértelo con .read_text()) — recomendado < 10 000 caracteres
  2. Escribe un script que reciba una pregunta por input() y responda usando context stuffing
  3. Añade al prompt la instrucción "si no está en el documento, di 'no está en el apunte'"
  4. Haz 3 preguntas: una que SÍ esté en el apunte, una que NO esté, y una ambigua. Documenta qué responde en cada caso
  5. Imprime los tokens usados y el coste estimado en cada llamada

Código de inicio:

import os
from pathlib import Path
from anthropic import Anthropic
from dotenv import load_dotenv

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

# 1. Carga tu apunte
apunte = Path("mi_apunte.md").read_text(encoding="utf-8")

def preguntar(pregunta: str) -> str:
    prompt = f"""Responde usando SOLO el siguiente apunte. Si la información no está, di "no está en el apunte".

APUNTE:
{apunte}

PREGUNTA: {pregunta}"""

    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}]
    )
    print(f"\nPregunta: {pregunta}")
    print(f"Respuesta: {resp.content[0].text}")
    print(f"Tokens in/out: {resp.usage.input_tokens}/{resp.usage.output_tokens}")
    return resp.content[0].text

# 2. Haz las 3 preguntas
preguntar("¿Cuál es la idea principal del apunte?")          # SÍ
preguntar("¿Qué opina Juan sobre el fútbol?")                # NO
preguntar("¿Hay algo relacionado con programación?")         # Ambigua

Bonus: añade un contador global de tokens gastados durante la sesión y calcula el coste total.
