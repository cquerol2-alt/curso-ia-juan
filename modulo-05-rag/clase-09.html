🧭 Filosofía: sin tracing, en producción estás ciego. Con tracing, cada queja de un usuario se convierte en un trace reproducible que puedes abrir y corregir. La diferencia entre "ayer fallaba algo" y "aquí está exactamente qué chunk devolvió mal el retriever".

📖 Qué es un trace (y por qué importa)

Una query del usuario dispara varias operaciones en tu backend. Un trace es el árbol completo de esas operaciones, con tiempos, inputs y outputs de cada paso.

Ejemplo de trace RAG 
trace: "user asked: cómo funciona el prompt caching"
├─ span retrieve (k=20)                     45 ms
│   input: { query: "cómo funciona el prompt caching" }
│   output: 20 candidates
├─ span rerank (haiku)                     380 ms   $0.001
│   input: 20 candidates
│   output: top 3
└─ span answer (sonnet)                   1120 ms   $0.014
    input: prompt + 3 chunks
    output: "El prompt caching cachea..."
    tokens: { prompt: 2850, completion: 180 }

Con esto puedes responder preguntas como: ¿cuánto cuesta la media de queries?, ¿qué queries tardan más de 3s?, ¿qué chunk es el más recuperado (quizá erróneamente)?.

💻 Práctica: deploy + observability (60 min)

Paso 1 · Cuenta Langfuse (gratis hasta 50k observaciones/mes)

  1. Crea cuenta en https://cloud.langfuse.com
  2. Crea un proyecto nuevo "rag-chat"
  3. Copia la public key y la secret key

Paso 2 · Instrumenta el backend Python

Bash 
pip install langfuse
Python — rag_pipeline.py (instrumentado) 
import os
from anthropic import Anthropic
from langfuse.decorators import observe, langfuse_context
from langfuse import Langfuse

os.environ["LANGFUSE_PUBLIC_KEY"] = "pk-lf-..."
os.environ["LANGFUSE_SECRET_KEY"] = "sk-lf-..."
os.environ["LANGFUSE_HOST"] = "https://cloud.langfuse.com"

langfuse = Langfuse()
client = Anthropic()

@observe(name="retrieve")
def retrieve(query: str, k: int = 20):
    # ... tu código ChromaDB
    results = col.query(query_embeddings=[embed([query])[0]], n_results=k)
    langfuse_context.update_current_observation(
        input={"query": query, "k": k},
        output={"n_candidates": k},
    )
    return [...]

@observe(name="rerank")
def rerank(query, candidates, n=3):
    # ... tu código con Haiku
    langfuse_context.update_current_observation(
        input={"query": query, "n_candidates": len(candidates)},
        output={"top_ids": [c["id"] for c in top]},
    )
    return top

@observe(name="answer", as_type="generation")  # marca este span como llamada LLM
def answer(query):
    candidates = retrieve(query)
    top = rerank(query, candidates)
    ctx = "\n---\n".join([f"[{c['source']}] {c['text']}" for c in top])

    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="Responde usando SOLO los chunks dados.",
        messages=[{"role": "user", "content": f"{ctx}\n\nPregunta: {query}"}],
    )
    text = resp.content[0].text

    # Langfuse registra tokens y coste si le das el modelo y usage
    langfuse_context.update_current_observation(
        input={"query": query},
        output=text,
        model="claude-sonnet-4-6",
        usage={
            "input": resp.usage.input_tokens,
            "output": resp.usage.output_tokens,
        },
    )
    return text

Paso 4 · Deploy del backend Python (Render o Railway)

Vercel no corre Python persistente ni ChromaDB. Opciones:

  • Render.com (gratis con sleep) — Dockerfile con uvicorn + ChromaDB local
  • Railway ($5/mes, sin sleep) — igual pero con volumen persistente
  • Fly.io — más control, curva más empinada
Dockerfile (para Render) 
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
# Asume que los apuntes indexados están en ./chroma_db (commitea el volumen)
CMD ["uvicorn", "rag_api:app", "--host", "0.0.0.0", "--port", "8000"]

Paso 5 · Verifica las trazas en Langfuse

  1. Haz 5 queries desde el chat en producción (URL de Vercel)
  2. Abre cloud.langfuse.com → tu proyecto → Traces
  3. Clic en una traza — verás el árbol retrieve → rerank → answer con tiempos, tokens y coste
  4. Filtra por "tokens > 3000" o "latency > 2000ms" para detectar outliers

Paso 6 · Añade scoring manual (thumbs up/down)

En el frontend, añade botones 👍 / 👎 que llamen a /api/feedback. En el backend, registra con langfuse.score(trace_id, name="thumb", value=1|0). Así tienes métricas de calidad en producción sin tener que montar una eval suite manualmente.

📊 Qué mirar el primer día después del deploy

MétricaUmbral sanoQué hacer si falla
Latency p95< 3sBaja k del retrieve, reduce max_tokens, añade cache
Coste medio por query< $0.03Usa Haiku en rerank, corta chunks más pequeños
Error rate< 1%Revisa timeouts, añade retries con backoff
Thumbs-down rate< 10%Toma los traces con 👎 y conviértelos en eval set
Los traces con 👎 son oro puro: cada uno es un ejemplo real de fallo con su contexto completo. Añádelos a tu eval dataset y conviértete en antifragile.
