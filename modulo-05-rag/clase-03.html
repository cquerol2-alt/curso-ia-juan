Clase 03: Agentic search — el modelo decide qué buscar

📅 Semana 6 — Miércoles | ⏱️ 55 minutos

🤖 Por qué agentic search es la arquitectura que más usa la industria en 2026

En la clase 02 viste que el context stuffing gana cuando el documento cabe en la ventana. Pero, ¿qué pasa cuando tienes 100 PDFs, o cuando los datos cambian cada hora, o cuando no sabes de antemano qué parte necesitas?

La respuesta moderna no es RAG clásico por defecto. Es agentic search: le das al modelo un puñado de tools (search_docs, read_file, web_search, run_sql) y él decide qué invocar, con qué argumentos, y cuándo parar. Así funcionan Claude Code, Cursor, Perplexity y la gran mayoría de agentes serios que estás usando en 2026.

En esta clase montarás tu primer agente con tool use nativo de Anthropic — sin LangChain, sin wrappers, sin magia. Solo el SDK oficial, un bucle while, y funciones Python normales. Al final tendrás un agente que busca respuestas en los apuntes de clase de Juan por iniciativa propia.

📖 Lectura complementaria (15 min): Tool use with Claude — Anthropic Docs. Es la referencia oficial. Entiende tool_use, tool_result y el bucle stop_reason == "tool_use" antes de escribir código.

🧠 Conceptos clave

1. Tool use: el modelo pide, tu código ejecuta

Tool use (también llamado function calling) es un contrato muy simple:

  1. Le mandas al modelo una descripción de qué herramientas tiene disponibles (nombre, qué hace, qué parámetros acepta).
  2. Si el modelo decide que necesita llamar una herramienta, no la ejecuta él — devuelve un bloque tool_use con el nombre y los argumentos.
  3. Tu código ejecuta la función real y le devuelve el resultado en un mensaje tool_result.
  4. El modelo ve el resultado y continúa: puede volver a llamar otra tool, o responder al usuario.

Este bucle se llama patrón ReAct (Reason + Act). Es la base de cualquier agente LLM serio.

2. Anatomía de una tool

Una tool para la API de Anthropic es un dict con 3 campos:

tool_buscar = {
    "name": "search_docs",                    # identificador único
    "description": (
        "Busca en los apuntes de Juan. Devuelve los 3 fragmentos más "
        "relevantes con su archivo de origen. Usa esta tool SIEMPRE "
        "que la pregunta mencione clase, módulo o concepto del curso."
    ),                                        # lo más importante del dict
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "Texto libre a buscar (p. ej. 'tool use')"
            }
        },
        "required": ["query"]
    }
}
⚠️ La description es el 80% del trabajo. Si es vaga, el modelo llamará a la tool cuando no toca o con argumentos raros. Escríbela como si se la dieras a un becario: "haz X, devuelve Y, úsala cuando Z".

3. Bucle agéntico (pseudocódigo)

messages = [{"role": "user", "content": pregunta}]
while True:
    resp = client.messages.create(model=MODEL, tools=TOOLS, messages=messages, ...)
    if resp.stop_reason != "tool_use":
        return resp.content[0].text            # respuesta final

    # 1) guarda lo que dijo el modelo (puede incluir varios tool_use a la vez)
    messages.append({"role": "assistant", "content": resp.content})

    # 2) ejecuta cada tool_use y añade su tool_result
    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            resultado = TOOL_IMPL[block.name](**block.input)
            tool_results.append({
                "type": "tool_use_id",           # ⚠️ es tool_use_id, no id
                "tool_use_id": block.id,
                "content": resultado
            })
    messages.append({"role": "user", "content": tool_results})
Pon siempre un max_turnos (8–12) — si no, una tool rota puede hacer loop infinito y te arruina la API bill.

4. Agentic search vs context stuffing

Criterio Context stuffing Agentic search
Tamaño del corpus< 200K tokensIlimitado
Frescura de los datosEstática (lo que metes)Dinámica (tools en vivo)
Coste por queryPaga doc entero siemprePaga solo lo que lee
Latencia1 turno (1–2 s con caché)N turnos (3–15 s)
DebugFácil (un solo prompt)Complejo (pasos encadenados)
Úsalo cuandoUn doc, muchas queriesCorpus vivo, info no predecible

Paso 2 — Tu primera tool: get_current_time

Empieza por la tool más tonta del mundo — una que no necesita argumentos — para ver el bucle funcionando.

import os
from datetime import datetime
from zoneinfo import ZoneInfo
from dotenv import load_dotenv
import anthropic

load_dotenv()
client = anthropic.Anthropic()
MODEL = "claude-sonnet-4-6"

def get_current_time():
    return datetime.now(ZoneInfo("Europe/Madrid")).isoformat(timespec="seconds")

TOOLS = [{
    "name": "get_current_time",
    "description": "Devuelve la fecha y hora actual en Madrid en formato ISO.",
    "input_schema": {"type": "object", "properties": {}}
}]

resp = client.messages.create(
    model=MODEL,
    max_tokens=512,
    tools=TOOLS,
    messages=[{"role": "user", "content": "¿Qué hora es ahora mismo en Madrid?"}]
)
print(resp.stop_reason)    # 'tool_use'
print(resp.content)
En esta primera llamada el modelo no responde — te pide que ejecutes la tool. Ese es el punto.

Paso 3 — Cerrar el bucle

from anthropic.types import ToolUseBlock

def run_once(pregunta: str, max_turnos: int = 6) -> str:
    messages = [{"role": "user", "content": pregunta}]
    for _ in range(max_turnos):
        resp = client.messages.create(
            model=MODEL, max_tokens=1024, tools=TOOLS, messages=messages
        )
        if resp.stop_reason != "tool_use":
            # respuesta final: saca el primer bloque de texto
            return next(b.text for b in resp.content if getattr(b, "text", None))

        messages.append({"role": "assistant", "content": resp.content})
        tool_results = []
        for block in resp.content:
            if isinstance(block, ToolUseBlock) and block.name == "get_current_time":
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": get_current_time()
                })
        messages.append({"role": "user", "content": tool_results})
    return "[agente agotó max_turnos]"

print(run_once("¿Qué día es hoy?"))

Paso 4 — Tool search_docs sobre los apuntes

Crea una carpeta apuntes/ con 3-5 ficheros .md (puedes copiar texto de esta clase y de la clase 02). Luego añade esta tool:

from pathlib import Path

APUNTES_DIR = Path("apuntes")

def search_docs(query: str, max_results: int = 3) -> str:
    """Búsqueda por substring case-insensitive. Rudimentaria, pero suficiente."""
    q = query.lower()
    hits = []
    for p in APUNTES_DIR.rglob("*.md"):
        for i, line in enumerate(p.read_text(encoding="utf-8").splitlines(), 1):
            if q in line.lower():
                hits.append(f"[{p.name}:L{i}] {line.strip()}")
                if len(hits) >= max_results:
                    return "\n".join(hits)
    return "\n".join(hits) or "Sin resultados."

TOOLS.append({
    "name": "search_docs",
    "description": (
        "Busca por substring en los apuntes del curso (carpeta ./apuntes). "
        "Devuelve hasta `max_results` líneas con formato [archivo:Lnn]. "
        "Úsala SIEMPRE que la pregunta hable de clase, módulo, concepto o "
        "técnica del curso."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "Texto a buscar"},
            "max_results": {"type": "integer", "default": 3}
        },
        "required": ["query"]
    }
})

Paso 5 — Dispatcher genérico y run_agent

Hasta ahora el bucle sabía ejecutar una tool hardcodeada. Generalízalo:

TOOL_IMPL = {
    "get_current_time": lambda **_: get_current_time(),
    "search_docs": lambda query, max_results=3: search_docs(query, max_results),
}

def run_agent(pregunta: str, max_turnos: int = 8) -> str:
    messages = [{"role": "user", "content": pregunta}]
    for turno in range(max_turnos):
        resp = client.messages.create(
            model=MODEL, max_tokens=1024, tools=TOOLS, messages=messages
        )
        if resp.stop_reason != "tool_use":
            return next((b.text for b in resp.content if getattr(b, "text", None)), "")

        messages.append({"role": "assistant", "content": resp.content})
        results = []
        for block in resp.content:
            if isinstance(block, ToolUseBlock):
                try:
                    out = str(TOOL_IMPL[block.name](**block.input))
                except Exception as e:
                    out = f"ERROR: {e}"
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": out
                })
                print(f"  [turno {turno}] {block.name}({block.input}) → {out[:80]}…")
        messages.append({"role": "user", "content": results})
    return "[agente agotó max_turnos sin respuesta final]"

print(run_agent("¿En qué clase explicamos prompt caching y qué multiplicador tiene la primera llamada?"))
Observa los prints: el modelo llamará a search_docs("prompt caching") por sí solo, leerá los fragmentos, y te responderá citando la clase. Tú no le dijiste qué buscar. Eso es agentic search.

🎓 Resumen y siguientes pasos

  • El modelo no ejecuta tools: pide, tu código ejecuta, tú devuelves tool_result.
  • La calidad del agente depende un 80% de las descriptions de las tools.

  • Agentic search gana a stuffing cuando el corpus es grande o dinámico.
