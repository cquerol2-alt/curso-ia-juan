Mind Map: Observability y Debugging de Agentes con LangSmith

← Volver a la Clase 05
Por qué un agente necesita observability ▼
Caja negra peligrosa: un agente ejecuta una trayectoria multi-paso; mirar solo la respuesta final esconde fallos.
Fallo 1 — Bucles: el agente llama la misma tool una y otra vez con argumentos parecidos.
Fallo 2 — Tool equivocada: usa buscar_web cuando debería usar consultar_bbdd e inventa la respuesta.
Fallo 3 — Tokens fugados: devuelve respuesta correcta pero se come 40k tokens innecesarios.
Sin tracing no los ves. Observability = ver la trayectoria completa para cazarlos.
Conceptos clave: project, run, trace, span ▼
Project: carpeta lógica donde agrupas runs (ej: "agente-ria-prod" vs "agente-ria-dev").
Run: una ejecución completa del agente — una conversación o una llamada.
Trace: el árbol de pasos de un run, con timings y tokens. Un flamegraph aplicado a tu agente.
Span: cada nodo individual — una llamada LLM, un tool call, una función decorada.
Mismo modelo mental que Datadog, Honeycomb, Jaeger — aplicado a LLMs.
Setup en 10 líneas ▼
Cuenta + API key: smith.langchain.com → Settings → API Keys (lsv2_pt_...).
4 env vars: LANGSMITH_TRACING=true, LANGSMITH_API_KEY, LANGSMITH_PROJECT, ANTHROPIC_API_KEY.
Wrap del cliente: client = wrap_anthropic(Anthropic()) instrumenta todas las messages.create.
Decorador entrypoint: @traceable en la función principal del agente crea el span padre.
Decoradores anidados: funciones auxiliares con @traceable aparecen como spans hijos — útil para parsing bugs disfrazados.
Qué ves en un trace de LangSmith ▼
Waterfall: cascada horizontal donde cada barra es un span, la longitud = duración.
Input y output exactos: el prompt completo enviado al modelo, no una versión truncada.
Tokens in/out y coste estimado por span.
Metadatos: modelo (claude-sonnet-4-6), temperatura, stop_reason.
Agregados de proyecto: latencia p50/p95, coste medio por run, error rate por tool.
Workflow de debugging cuando algo falla ▼
Paso 1 — Localizar el run: filtra por proyecto, fecha y texto del input.
Paso 2 — Mirar el waterfall: ¿cuántos spans? ¿dónde está la latencia? ¿falta una tool esperada?
Paso 3 — Leer el prompt exacto del span LLM — casi siempre el bug está en la descripción de la tool o en el system prompt, no en el modelo.
Paso 4 — Compare runs: side-by-side del run fallido vs el corregido para validar el fix.
Paso 5 — Add to dataset: el input del bug se convierte en test de regresión permanente.
🚩 Red flags siempre bug: misma tool 3+ veces (loop), stop_reason=max_tokens, tool call con input {}, span LLM > 20 s.
De debugging a evals: datasets, experiments y alternativas ▼
Dataset: golden set de 30-50 inputs con output esperado. Se alimenta desde traces reales con "Add to dataset".
Experiment: corre tu agente contra el dataset + evaluators (funciones que devuelven un score) — el equivalente a pytest pero con LLMs.
Comparar versiones: corre el mismo dataset contra v1 y v2 → tabla diff: cuántos mejoraron, cuántos empeoraron.
Langfuse: alternativa europea, open-source, self-hostable. Mismo modelo mental.
Arize Phoenix: open-source, habla OpenTelemetry, ideal si el stack ya usa OTel.
Weave (W&B): tiene sentido si el equipo ya vive en Weights & Biases.
Recomendación: empieza con LangSmith cloud (5 min y tracing); cambia a Langfuse/Phoenix self-hosted cuando tengas datos sensibles.
