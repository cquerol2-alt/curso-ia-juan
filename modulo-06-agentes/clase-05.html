Clase 05: Observability y Debugging de Agentes con LangSmith

Vídeo Explicativo

🔗 Conexión con las clases anteriores: En M5 c07 aprendiste a medir un RAG con Ragas. En M6 c01–c04 has construido un agente que razona y llama herramientas. Pero cuando un agente de producción falla — o tarda 40 segundos — necesitas ver qué está pasando por dentro: qué herramientas llamó, con qué argumentos, en qué orden, cuántos tokens se comió cada paso. Aquí es donde entra observability.

Part A: Por qué un agente sin observability es una caja negra peligrosa

Un LLM normal es una función: entra prompt, sale texto. Si falla, relees el prompt y listo. Un agente, en cambio, ejecuta una trayectoria: piensa → llama herramienta A → lee resultado → piensa otra vez → llama herramienta B → …. Cuando algo va mal, tienes que reconstruir ese árbol de pasos. Sin herramientas, significa leer logs de texto plano a las 2 de la madrugada.

⚠️ Los tres fallos silenciosos de un agente:
  1. Bucles infinitos — llama la misma herramienta una y otra vez con argumentos parecidos.
  2. Herramientas equivocadas — usa buscar_web cuando debería usar consultar_bbdd, y acaba inventando la respuesta.
  3. Tokens fugados — devuelve la respuesta correcta pero se ha comido 40.000 tokens porque ha re-leído 10 veces el mismo documento.
Ninguno de estos tres fallos aparece si solo miras la respuesta final. Sólo se ven con tracing.

Qué resuelve una herramienta de observability

Pregunta Sin observability Con LangSmith
¿Por qué tardó 42 s? "Ni idea, pega el log en un bloc de notas" Ves un waterfall: la llamada 3 tardó 28 s porque la API externa estaba lenta
¿Cuánto cuesta una conversación típica? Calculas a mano sumando logs Panel con coste medio por run, p50/p95
¿Qué herramienta falla más? Tendrías que parsear cientos de logs Tabla agregada con error rate por tool
¿Mi prompt nuevo es mejor que el viejo? Pruebas manual en 5 ejemplos Dataset + experiment: corres los 50 casos automáticamente y comparas
¿Mi agente se está degradando? Te enteras cuando un cliente se queja Alertas sobre drift de latencia, coste o tasa de error

LangSmith (de los creadores de LangChain) es la herramienta más popular del ecosistema para esto. No requiere usar LangChain — funciona con cualquier código Python/TS, solo hay que envolver las funciones.

Part B: LangSmith 101 — runs, traces y spans

Para usar LangSmith bien hay que entender 4 conceptos. Son los mismos que en cualquier sistema de observability moderno (Datadog, Honeycomb, Jaeger), solo que aplicados a LLMs y agentes.

Concepto Qué es Ejemplo en un agente
Project Carpeta lógica donde se agrupan runs "agente-ria-qa-prod", "agente-ria-qa-dev"
Run Una ejecución completa del agente (una conversación o una llamada) "Usuario pregunta: ¿cómo se monta una paella para 40?"
Trace El árbol de pasos de un run, con timings y tokens LLM → tool_call(buscar_recetas) → LLM → tool_call(calcular_ingredientes) → LLM final
Span Cada nodo individual del trace (una llamada LLM, un tool call, una función) La llamada a calcular_ingredientes(personas=40, receta="paella")

La analogía mental: un trace es un flamegraph

Si Juan ha tocado DevTools en el navegador, un trace de LangSmith es muy parecido a la pestaña Performance: una cascada horizontal donde cada barra es un paso, la longitud indica duración, y puedes expandir para ver detalles. En LangSmith cada barra es un span y al expandirlo ves el prompt exacto, la respuesta, los tokens y el coste.

💡 Regla práctica: el 80% del debugging de un agente se resuelve abriendo el trace y leyendo el primer span que falló. El otro 20% es entender por qué el LLM decidió llamar esa herramienta — y para eso miras el prompt completo del span anterior, no solo el output.

Part C: Setup — tracing de un agente en 10 líneas

LangSmith tiene un plan gratuito generoso (5.000 traces/mes). El setup es casi cero: una variable de entorno y un decorador. Funciona con el SDK de Anthropic directamente, sin LangChain.

Paso 1: cuenta y API key

  1. Ir a smith.langchain.com y crear cuenta.
  2. En Settings → API Keys, generar una key (lsv2_pt_...).
  3. Guardarla en .env junto con la de Anthropic.

Paso 2: variables de entorno

LANGSMITH_TRACING=true LANGSMITH_API_KEY=lsv2_pt_... LANGSMITH_PROJECT=agente-ria-qa-dev ANTHROPIC_API_KEY=sk-ant-...

Paso 3: instrumentar el cliente

pip install langsmith anthropic python-dotenv
from dotenv import load_dotenv from langsmith import traceable from langsmith.wrappers import wrap_anthropic from anthropic import Anthropic load_dotenv() # wrap_anthropic inyecta tracing automático en todas las llamadas client = wrap_anthropic(Anthropic()) TOOLS = [ { "name": "calcular_ingredientes", "description": "Calcula cantidades para una receta según nº de comensales", "input_schema": { "type": "object", "properties": { "receta": {"type": "string"}, "personas": {"type": "integer"} }, "required": ["receta", "personas"] } } ] def calcular_ingredientes(receta: str, personas: int) -> dict: # stub de ejemplo return {"arroz_g": personas * 100, "caldo_l": personas * 0.25} @traceable(name="agente_qa_ria") def ejecutar_agente(pregunta: str) -> str: mensajes = [{"role": "user", "content": pregunta}] for _ in range(10): # max 10 pasos resp = client.messages.create( model="claude-sonnet-4-6", max_tokens=1024, tools=TOOLS, messages=mensajes, ) if resp.stop_reason == "end_turn": return resp.content[0].text # el modelo pidió tool_use tool_use = next(b for b in resp.content if b.type == "tool_use") resultado = calcular_ingredientes(**tool_use.input) mensajes.append({"role": "assistant", "content": resp.content}) mensajes.append({ "role": "user", "content": [{ "type": "tool_result", "tool_use_id": tool_use.id, "content": str(resultado) }] }) return "(sin respuesta tras 10 pasos)" print(ejecutar_agente("Ingredientes de paella para 40 personas"))

Con @traceable alrededor del entrypoint y wrap_anthropic alrededor del cliente, cada llamada al LLM y cada tool call aparece en LangSmith como un span hijo. No hay que añadir nada más.

Paso 4: abrir el trace

Al ejecutar el script, la consola imprime un enlace estilo https://smith.langchain.com/o/.../r/abc-123. Al pinchar verás:

  • Árbol completo de spans (el padre agente_qa_ria y sus hijos LLM + tool).
  • Input y output de cada span (el prompt exacto, no una versión truncada).
  • Tiempo por span (el waterfall).
  • Tokens in/out y coste estimado por span.
  • Metadatos: modelo, temperatura, stop_reason.
ℹ️ Decoradores anidados. Si tienes funciones auxiliares (parsing, validación), decorarlas también con @traceable las hace aparecer como spans. Útil cuando un parsing bug está disfrazado de "el LLM da mala respuesta".

Part D: Workflow de debugging cuando tu agente hace cosas raras

Hay una secuencia mental que sirve para 9 de cada 10 bugs. Vamos a aplicarla sobre un caso típico.

🎬 Escenario: un cliente reporta que el agente de RIA respondió "no tengo información" a la pregunta "¿cuánto cuesta montar una paella para 40?". Tú sabes que la tool calcular_ingredientes sí sabe responder. ¿Qué falló?

Paso 1 — localizar el run

En LangSmith, filtras por project = prod, date = hoy, y buscas texto "paella para 40" en el input. Un click y tienes el trace abierto.

Paso 2 — mirar el waterfall

Si el trace tiene un único span LLM y ningún tool call, ya lo tienes: el modelo no decidió llamar la herramienta. El bug está en el prompt del sistema o en la descripción de la tool. No es bug del modelo.

Paso 3 — leer el prompt completo del span LLM

Abres el span, miras el input. Ahí ves que la tool description que tu código pasa es: "Calcula cantidades para una receta". Demasiado vago. El modelo no asoció "cuánto cuesta montar una paella" con "calcula cantidades". Cambias la descripción a "Calcula cantidades e ingredientes necesarios para preparar una receta (paella, arroces, etc.) según número de comensales" y reejecutas.

Paso 4 — usar "Compare runs"

Con el run viejo y el nuevo abiertos en pestañas distintas, LangSmith tiene botón Compare: te muestra side-by-side qué cambió. Si el nuevo llama la tool correctamente, el bug estaba en la descripción. Si sigue igual, hay algo más.

Paso 5 — guardar el caso como regresión

Cuando lo arreglas, añades ese input al dataset de LangSmith (botón Add to dataset). Así, la próxima vez que cambies el prompt, puedes correr ese dataset contra la versión nueva y verificar que "paella para 40" sigue funcionando. Esto es la bisagra entre observability y evals.

⚠️ Red flags en traces que son siempre bug:
  • El mismo tool call repetido 3+ veces con inputs parecidos (loop).
  • Span LLM con stop_reason = "max_tokens" (el output se cortó).
  • Tool call con input {} vacío (el modelo no pudo extraer argumentos).
  • Latencia > 20 s en un solo span LLM (modelo o red, no tu código).

Part E: De debugging a evals — datasets y experiments

LangSmith no es solo un visor de traces. Cuando acumulas runs reales puedes convertirlos en datasets y correr experiments: la evolución natural desde "arreglar un bug" hacia "medir si mi nueva versión es mejor".

Datasets: ejemplos que no quieres romper nunca

  • Creas un dataset "golden_qa_ria" con 30–50 inputs reales.
  • Para cada input anotas el output esperado o un criterio ("debe llamar la tool X").
  • Puedes añadir inputs desde cualquier trace con un click (Add to dataset).

Experiments: correr tu agente contra el dataset entero

from langsmith import Client from langsmith.evaluation import evaluate client = Client() def eval_llamo_tool_correcta(run, example): tool_names = [s.name for s in run.outputs.get("tool_calls", [])] esperado = example.outputs["tool_esperada"] return {"key": "tool_correcta", "score": 1 if esperado in tool_names else 0} results = evaluate( lambda inputs: {"answer": ejecutar_agente(inputs["pregunta"])}, data="golden_qa_ria", evaluators=[eval_llamo_tool_correcta], experiment_prefix="cambio-descripcion-tools", )

Al acabar, en LangSmith verás una tabla comparativa: cada ejemplo, qué respondió el agente, qué score sacó. Si corres el mismo dataset contra dos versiones de tu código (v1 y v2), te da el diff: cuántos ejemplos mejoraron, cuántos empeoraron, cuáles son nuevos fallos.

💡 Enlace con M5 c07. Los evaluators aquí son igual que los de Ragas: funciones que reciben el output del agente y devuelven un score. La diferencia es que en RAG evaluabas la respuesta (faithfulness), aquí evalúas la trayectoria (¿llamó la tool correcta? ¿en cuántos pasos?).

Alternativas a LangSmith

Herramienta Ecosistema Cuándo tiene sentido
LangSmith LangChain / standalone Default razonable. Setup mínimo, UI muy pulida, plan free generoso.
Arize Phoenix Open-source, OTel-compatible Si quieres self-hosted o ya usas OpenTelemetry en el resto del stack.
Weave (W&B) Weights & Biases Si tu equipo ya vive en W&B por ML tradicional.
Langfuse Open-source, self-hostable Alternativa europea a LangSmith. Buen plan free cloud + opción on-prem.
ℹ️ Recomendación para Juan. Empieza con LangSmith cloud — 5 minutos y estás tracing. Cuando hagas un proyecto donde no puedas mandar datos a servicios externos (datos sanitarios, legal, datos de clientes sensibles), te pasas a Langfuse self-hosted o Phoenix. La conceptualización (runs, traces, spans) es idéntica en las cuatro.

Ejercicios prácticos

🛠 Ejercicio 1: Tu primer trace

Objetivo: tener un agente tuyo instrumentado y ver el trace en LangSmith.

  1. Crea cuenta en smith.langchain.com y obtén API key.
  2. Añade LANGSMITH_* al .env.
  3. Copia el código de Part C, adáptalo con una tool tuya (puede ser la misma calcular_ingredientes u otra simple tipo buscar_precio_mercadona).
  4. Ejecuta con 3 preguntas distintas:
    • Una que debería disparar la tool.
    • Una que no necesita la tool (ej: "hola").
    • Una ambigua.
  5. Abre los 3 traces en LangSmith y anota en un doc:
    • ¿Cuántos spans tiene cada uno?
    • ¿Cuál fue la latencia total?
    • ¿Tokens in/out y coste estimado?

Entregable: capturas de los 3 traces + tabla con latencia/tokens/coste.

🔍 Ejercicio 2: Debugging guiado

Objetivo: usar el workflow de Part D para encontrar un bug real.

  1. Parte de tu agente del ejercicio 1.
  2. Sabotea la descripción de una tool: cámbiala por algo muy vago ("Hace una operación útil").
  3. Ejecuta 5 preguntas que deberían llamarla.
  4. En LangSmith:
    • ¿Cuántas de las 5 el agente llamó la tool correcta?
    • Abre un trace fallido y lee el span LLM: ¿por qué el modelo decidió no llamarla?
  5. Corrige la descripción (versión buena). Re-ejecuta las 5 mismas preguntas.
  6. Usa Compare runs para ver el diff.

Entregable: captura de la comparación + 1 párrafo explicando qué cambió en el comportamiento del modelo.

📊 Ejercicio 3: Dataset y mini experiment

Objetivo: convertir runs reales en un eval reproducible.

  1. En LangSmith, crea un dataset vacío llamado agente-juan-v1.
  2. Añade 5 inputs a mano (o desde traces anteriores con Add to dataset).
  3. Para cada uno anota en Reference outputs el tool esperado (ej: {"tool_esperada": "calcular_ingredientes"}).
  4. Escribe un evaluator que devuelva 1 si el agente llamó la tool esperada, 0 si no.
  5. Corre evaluate() con la versión buena del agente (del ejercicio 2).
  6. Cambia el modelo de claude-sonnet-4-6 a claude-haiku-4-5 y vuelve a correr como experiment 2.

Entregable: captura del panel de experiments comparando los dos modelos + 2 líneas sobre qué recomendarías a un cliente basándote en score vs coste.

Resumen de la clase

  • Un agente sin observability es una caja negra. Los 3 fallos que solo se ven trazando: bucles, tool equivocada, tokens fugados.
  • Conceptos clave: project → run → trace → span. Idénticos a cualquier APM moderno, aplicados a LLMs.
  • Setup LangSmith = 10 líneas: wrap_anthropic() en el cliente + @traceable en el entrypoint.
  • Workflow de debugging: localizar run → leer waterfall → leer prompt exacto del span → comparar runs → guardar como regresión.
  • Datasets + experiments cierran el bucle: runs reales → golden set → mides si cada cambio mejora o empeora.
  • Alternativas si no puedes usar cloud: Langfuse (self-hosted), Phoenix (OTel), Weave (W&B).
