📖 Resumen: Los Conceptos que Necesitas

Parte A: ¿Por Qué Docker?

"Works on my machine" es el problema más viejo de la programación.

  • Tu app funciona perfecto en tu laptop (Python 3.11, PostgreSQL 15, Redis 7.0)
  • Se la pasas al equipo: "Necesito Python 3.10, pero tú tienes 3.9..."
  • Se la pasas a producción: diferente SO, diferentes librerías instaladas → CRASH

Docker es como un contenedor de envío. Igual que un contenedor de carga funciona en cualquier barco, cualquier camión, cualquier tren, un contenedor Docker funciona en cualquier máquina, cualquier servidor, cualquier cloud.

Dato clave: El 90%+ de los job offers para "AI Engineer" o "ML Engineer" requieren experiencia con Docker. No es opcional; es tabla rasa en el mercado laboral.

Parte B: Conceptos Clave (Lo Justo para Ser Productivo)

Terminología Docker 
IMAGE (Imagen)
  └─ El blueprint, la receta. Un archivo inmutable que contiene:
     - Sistema operativo (Alpine, Ubuntu, etc.)
     - Runtime (Python, Node.js, etc.)
     - Dependencias de tu app
     - Tu código
  └─ Se construye una sola vez con `docker build`
  └─ Se puede copiar, compartir, reutilizar infinitas veces

CONTAINER (Contenedor)
  └─ Una instancia corriendo de la imagen. Como "el plato cocinado".
  └─ Aislado: tu app no interfiere con otras apps en la misma máquina
  └─ Ephemeral: muere cuando lo paras (`docker stop`)
  └─ Puedes tener 10 contenedores corriendo de la misma imagen

DOCKERFILE
  └─ El archivo de instrucciones para construir la imagen
  └─ Similar a un Makefile o package.json
  └─ Línea por línea: descarga base image → copia archivos → instala deps

DOCKER COMPOSE
  └─ Orquesta múltiples contenedores
  └─ Define: qué servicios necesitas (app, DB, cache)
  └─ Un solo comando: `docker compose up` levanta todo

REGISTRY (Docker Hub)
  └─ GitHub pero para images de Docker
  └─ Descargas images pre-hechas (python:3.11, nginx, postgres, etc.)
  └─ Subes tus propias images (privadas o públicas)

Nota importante: NO profundizamos en internals de Docker (namespaces, cgroups, layers). Solo lo que necesitas para ser productivo: escribir un Dockerfile, ejecutar contenedores, usarlos en desarrollo y producción.

Parte C: Tu Primer Dockerfile para Python/FastAPI

Imagina que tienes una app FastAPI. Así se ve el Dockerfile:

Dockerfile 
# 1. BASE IMAGE: Usa Python 3.11 en una versión slim (más pequeña)
FROM python:3.11-slim

# 2. DIRECTORIO DE TRABAJO dentro del contenedor
WORKDIR /app

# 3. COPIA el archivo de dependencias PRIMERO (para cachear en capas)
COPY requirements.txt .

# 4. INSTALA las dependencias (sin cache local para ahorrar espacio)
RUN pip install --no-cache-dir -r requirements.txt

# 5. COPIA el resto del código
COPY . .

# 6. EXPONE el puerto 8000 (documentación, no vinculante)
EXPOSE 8000

# 7. COMANDO para ejecutar la app
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

¿Qué es cada línea?

  • FROM: Descarga la imagen base (Python 3.11 slim, ~150MB)
  • WORKDIR: Crea /app y todas las siguientes operaciones ocurren ahí
  • COPY requirements.txt .: Copia requirements.txt del host al contenedor
  • RUN pip install: Ejecuta pip dentro del contenedor (instala numpy, fastapi, etc.)
  • COPY . .: Copia el código restante (main.py, etc.)
  • EXPOSE 8000: Documenta que la app escucha en puerto 8000
  • CMD: Comando por defecto cuando arranca el contenedor

.dockerignore: Así como .gitignore, exluye archivos que NO necesita la imagen:

.dockerignore 
venv/
__pycache__/
*.pyc
.git
.gitignore
.env
.DS_Store
node_modules/
dist/
build/

Construir y ejecutar:

bash 
# 1. Construir la imagen (se llama "mi-ai-app", versión "latest")
docker build -t mi-ai-app .

# 2. Ejecutar un contenedor basado en esa imagen
#    -p 8000:8000 = mapea puerto 8000 del contenedor al 8000 del host
#    --env-file .env = inyecta variables de entorno desde .env
docker run -p 8000:8000 --env-file .env mi-ai-app

# 3. En otra terminal, prueba la app
curl http://localhost:8000/docs

# 4. Ver qué contenedores están corriendo
docker ps

# 5. Parar el contenedor
docker stop <CONTAINER_ID>

Parte D: Docker Compose — App + Base de Datos

En la vida real, tu app no está sola. Necesita una BD, cache, etc. Para eso existe Docker Compose:

docker-compose.yml 
version: '3.9'

services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      CHROMA_HOST: chroma
      REDIS_URL: redis://redis:6379
    depends_on:
      - chroma
      - redis
    volumes:
      - ./:/app  # Recargar código en vivo (desarrollo)

  chroma:
    image: ghcr.io/chroma-core/chroma:latest
    ports:
      - "8001:8000"
    volumes:
      - chroma_data:/chroma/chroma

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data

volumes:
  chroma_data:
  redis_data:

¿Cómo se usa?

bash 
# Levanta TODOS los servicios (app, chroma, redis) con un comando
docker compose up

# En otra terminal, compila app en modo desarrollo
docker compose up app

# Para todo
docker compose down

# Borra volúmenes (BD y datos)
docker compose down -v

Parte E: Buenas Prácticas

  • Multi-stage builds: Construye en una etapa, copia solo lo necesario a la etapa final (reduce tamaño de imagen)
  • Usa imágenes -slim o -alpine: Bases pequeñas (50-150MB vs 800MB+)
  • Nunca metas secretos en la imagen: Usa --env-file .env o Docker secrets
  • Copia en el orden correcto: Dependencies antes de código (aprovecha el cacheo de capas)
  • Health checks: Dile a Docker cómo verificar que la app está viva
Dockerfile (Multi-stage Build - Avanzado) 
# Etapa 1: Construcción (más pesada)
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# Etapa 2: Runtime (solo lo esencial)
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]

💻 Ejercicios Prácticos

Ejercicio 1: Dockeriza un FastAPI "Hola Mundo" (Code-Along, 20 min)

Objetivo: Escribe un Dockerfile, construye la imagen y ejecuta un contenedor.

Paso 1: Crea la app FastAPI

main.py 
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI(title="Mi Primera App Dockerizada")

class Mensaje(BaseModel):
    texto: str

@app.get("/")
def root():
    return {"mensaje": "Hola desde Docker!"}

@app.post("/echo")
def echo(msg: Mensaje):
    return {"eco": msg.texto}

Paso 2: Crea requirements.txt

requirements.txt 
fastapi==0.104.1
uvicorn==0.24.0
pydantic==2.5.0

Paso 3: Escribe el Dockerfile

Dockerfile 
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Paso 4: Construye y ejecuta

bash 
# Construir
docker build -t mi-primer-app .

# Ejecutar
docker run -p 8000:8000 mi-primer-app

# En otra terminal, prueba
curl http://localhost:8000
# Respuesta: {"mensaje":"Hola desde Docker!"}

curl -X POST http://localhost:8000/echo \
  -H "Content-Type: application/json" \
  -d '{"texto":"Esto funciona!"}'
# Respuesta: {"eco":"Esto funciona!"}

Ejercicio 2: Mini-Challenge — Dockeriza Tu Chatbot (30 min)

Objetivo: Toma el chatbot del Module 3 Boss Battle, crea un Dockerfile y un docker-compose.yml con Redis.

Pasos:

  1. Crea una carpeta nueva: chatbot-dockerizado/
  2. Copia tu código del chatbot (chatbot.py o similar)
  3. Crea un main.py que exponga el chatbot como API FastAPI:
    main.py 
    from fastapi import FastAPI
from chatbot import ChatBot

app = FastAPI()
bot = ChatBot()

@app.post("/chat")
def chat(user_message: str):
    response = bot.responde(user_message)
    return {"response": response}
  4. Escribe el Dockerfile para el chatbot
  5. Crea un docker-compose.yml que levante:
    • Tu app (puerto 8000)
    • Redis (puerto 6379) para cachear respuestas
  6. Ejecuta: docker compose up
  7. Prueba: curl -X POST http://localhost:8000/chat?user_message="Hola"
