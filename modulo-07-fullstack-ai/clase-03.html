📚 Resumen de Conceptos

¿Por qué Autenticación en una Aplicación de IA?

Cuando construyes una app que consume APIs de IA (OpenAI, Anthropic, etc.), la autenticación es crítica:

  • Proteger API keys: Nunca exponer claves en el cliente. Las autenticación segura evita que usuarios vean tus credenciales.
  • Controlar costes: Cada request a una API cuesta dinero. Sin autenticación, cualquiera puede gastar tu presupuesto.
  • Personalización: Conocer quién es el usuario permite guardar históricos, preferencias, contexto personal.
  • Cumplimiento normativo: RGPD, CCPA y otras normativas requieren identificar y proteger datos de usuarios.

Auth.js (Anteriormente NextAuth.js)

¿Qué es?

Auth.js es un framework de autenticación de código abierto para aplicaciones web modernas (especialmente Next.js, pero también funciona en otros stacks). Proporciona:

  • Autenticación con proveedores OAuth2 (Google, GitHub, Discord, etc.)
  • Autenticación por Email/Contraseña
  • Gestión automática de sesiones (cookies seguras)
  • Middleware para proteger rutas
  • Callbacks para personalizar el flujo de autenticación

¿Cómo funciona?

  1. Usuario hace login: Selecciona un proveedor (ej: "Continuar con Google")
  2. Redirección OAuth: Se envía a Google, el usuario autoriza, Google devuelve un token
  3. Sesión creada: Auth.js crea una sesión segura (JWT o cookie) y la guarda en el servidor o base de datos
  4. Cliente recibe sesión: El navegador almacena la sesión (cookie segura). Siguientes requests la incluyen automáticamente
  5. Middleware valida: Cada request se valida; si la sesión es válida, el usuario accede; si no, se redirige a login

Proveedores Soportados

Auth.js soporta cientos de proveedores. Los más comunes:

  • Google: El más popular. Requiere OAuth 2.0 credentials en Google Cloud Console
  • GitHub: Perfecto para apps de desarrolladores. Crea OAuth App en GitHub Settings
  • Email/Password: Autenticación tradicional. Requiere base de datos para guardar usuarios
  • Credenciales personalizadas: Puedes crear tu propio proveedor

Sessions vs JWT: ¿Cuándo Usar Cada Una?

Aspecto Session (Cookie) JWT
Storage Servidor (base de datos) + Cookie en cliente Cliente (localStorage/sessionStorage)
Seguridad ✅ Más segura (no expone credenciales en cliente) ⚠️ Vulnerable a XSS si se guarda en localStorage
Escalabilidad Requiere base de datos centralizada ✅ Stateless, escalable sin base de datos
Revocación ✅ Inmediata (borrar de base de datos) ⚠️ Difícil (token sigue siendo válido hasta expirar)
Mejor para Apps monolíticas, Next.js, alta seguridad APIs REST, microservicios, apps móviles

Recomendación para este curso: Usa Sessions con Auth.js en Next.js. Es más simple y más segura para aplicaciones fullstack.

Rate Limiting: ¿Qué es y por qué es CRÍTICO en Apps de IA?

Definición

Rate Limiting es un mecanismo que controla cuántas requests puede hacer un usuario en un período de tiempo. Por ejemplo: "máximo 100 requests por minuto".

¿Por qué es crítico?

En aplicaciones de IA, cada request cuesta dinero:

  • OpenAI GPT-4: $0.03 por 1K tokens de entrada
  • Anthropic Claude: $0.80 por 1M tokens de entrada
  • Google Vertex AI: Varía, pero puede ser caro si hay tráfico masivo

Sin rate limiting, un atacante podría:

  • Hacer 10,000 requests en 1 segundo (DoS attack)
  • Gastar $10,000 en 1 minuto
  • Hacer que tu servicio sea inusable para otros usuarios

Rate Limiting evita esto limitando el daño.

Estrategias de Rate Limiting

Estrategia Descripción Ventajas Desventajas
Por Usuario Cada usuario tiene su cuota (ej: 1000 requests/día) ✅ Justo, personalizable por plan ⚠️ Requiere base de datos
Por IP Limitar por dirección IP origen ✅ Rápido, sin base de datos ⚠️ Fácil de evadir con proxies
Por API Key Cada app/cliente obtiene una clave con su cuota ✅ Flexible, seguro ⚠️ Requiere gestión de claves
Global Límite total para todo el servicio ✅ Muy simple ⚠️ Poco granular

Upstash Rate Limit: La Solución más Popular para Serverless

¿Qué es Upstash?

Upstash es una plataforma de base de datos Redis serverless en la nube. Su característica Rate Limiting es perfecta para:

  • Next.js + Vercel (integración muy simple)
  • Aplicaciones sin servidor (no necesitas tu propia infraestructura)
  • Control granular por usuario, IP o API key
  • Rápidísimo (Redis es ultrarrápido)

Plan Gratuito

Upstash ofrece:

  • 10,000 comandos/día (suficiente para desarrollo)
  • Database Redis gratuita
  • Sin tarjeta de crédito requerida

⚙️ Práctica Guiada: Autenticación + Rate Limiting en una Chat App

Objetivo: Crear una aplicación de chat segura que proteja tu API key, autentique usuarios y limite su uso.

Step 1: Instalar Auth.js

En tu proyecto Next.js 13+:

npm install next-auth

Auth.js ya está integrado en Next.js 13+. Si usas una versión anterior, también: npm install @auth/core

Step 2: Configurar Auth.js con Google OAuth

Crea app/api/auth/[...nextauth]/route.ts:

import NextAuth from 'next-auth';
import GoogleProvider from 'next-auth/providers/google';

const handler = NextAuth({
  providers: [
    GoogleProvider({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
  ],
  pages: {
    signIn: '/auth/signin',
  },
  callbacks: {
    async jwt({ token, account }) {
      if (account) {
        token.accessToken = account.access_token;
      }
      return token;
    },
    async session({ session, token }) {
      session.user.id = token.sub;
      return session;
    },
  },
});

export { handler as GET, handler as POST };
⚠️ Credenciales de Google:
  1. Ve a Google Cloud Console
  2. Crea un proyecto nuevo
  3. Habilita "Google+ API"
  4. Crea credenciales OAuth 2.0 (tipo "Web application")
  5. Añade Authorized Redirect URI: http://localhost:3000/api/auth/callback/google
  6. Copia Client ID y Client Secret a tu .env.local

Step 3: Crear un Middleware para Proteger Rutas

Crea middleware.ts en la raíz:

import { withAuth } from 'next-auth/middleware';

export const config = {
  matcher: [
    '/api/chat/:path*',  // Proteger la ruta de chat
    '/dashboard/:path*', // Proteger el dashboard
  ],
};

export default withAuth(function middleware(req) {
  // El middleware deja pasar si el usuario está autenticado
  // Si no, Auth.js automáticamente redirige a signin
});

Step 4: Proteger la Ruta de Chat API (Solo Usuarios Autenticados)

Crea app/api/chat/route.ts:

import { getServerSession } from 'next-auth';
import { NextRequest, NextResponse } from 'next/server';

export async function POST(req: NextRequest) {
  // 1. Verificar que el usuario está autenticado
  const session = await getServerSession();
  if (!session || !session.user) {
    return NextResponse.json(
      { error: 'No autenticado' },
      { status: 401 }
    );
  }

  const { message } = await req.json();
  const userId = session.user.id;

  // 2. Aquí vendría el rate limiting (Step 5)

  // 3. Llamar a OpenAI (API key está segura en servidor)
  const response = await fetch('https://api.openai.com/v1/chat/completions', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.OPENAI_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: 'gpt-4o-mini',
      messages: [{ role: 'user', content: message }],
    }),
  });

  const data = await response.json();
  return NextResponse.json({ reply: data.choices[0].message.content });
}

Step 5: Implementar Rate Limiting Simple (En Memoria)

Opción 1: Rate limiting básico en memoria (solo para desarrollo):

// lib/rateLimit.ts
const requestMap = new Map<string, number[]>();

export function checkRateLimit(
  userId: string,
  limit: number = 10,
  windowMs: number = 60000 // 1 minuto
): boolean {
  const now = Date.now();
  const requests = requestMap.get(userId) || [];

  // Filtrar requests dentro de la ventana de tiempo
  const recentRequests = requests.filter(time => now - time < windowMs);

  // Comprobar si se excedió el límite
  if (recentRequests.length >= limit) {
    return false; // Rate limit excedido
  }

  // Registrar nuevo request
  recentRequests.push(now);
  requestMap.set(userId, recentRequests);

  return true; // OK
}

Opción 2: Rate Limiting con Upstash (Producción):

npm install @upstash/ratelimit redis

// lib/rateLimit.ts
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const redis = new Redis({
  url: process.env.UPSTASH_REDIS_REST_URL!,
  token: process.env.UPSTASH_REDIS_REST_TOKEN!,
});

export const ratelimit = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(10, '1 m'), // 10 requests por minuto
});

export async function checkRateLimitUpstash(userId: string) {
  try {
    const { success } = await ratelimit.limit(userId);
    return success;
  } catch (error) {
    console.error('Rate limit error:', error);
    return false; // En caso de error, denegar
  }
}

Step 6: Combinar Auth + Rate Limit en la API

Actualiza app/api/chat/route.ts:

import { getServerSession } from 'next-auth';
import { checkRateLimitUpstash } from '@/lib/rateLimit';
import { NextRequest, NextResponse } from 'next/server';

export async function POST(req: NextRequest) {
  // 1. Autenticación
  const session = await getServerSession();
  if (!session?.user?.id) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  // 2. Rate Limiting
  const allowed = await checkRateLimitUpstash(session.user.id);
  if (!allowed) {
    return NextResponse.json(
      { error: 'Límite de requests excedido. Espera 1 minuto.' },
      { status: 429 } // Too Many Requests
    );
  }

  const { message } = await req.json();

  // 3. Llamar a OpenAI
  const response = await fetch('https://api.openai.com/v1/chat/completions', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.OPENAI_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: 'gpt-4o-mini',
      messages: [{ role: 'user', content: message }],
    }),
  });

  const data = await response.json();
  return NextResponse.json({ reply: data.choices[0].message.content });
}
