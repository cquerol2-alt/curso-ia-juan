🧠 Mind Map: Autenticación y Rate Limiting

Conceptos clave de seguridad en aplicaciones de IA — Módulo 7, Clase 03

🔐 Autenticación: Identificar Usuarios ▼
¿Por qué es crítica en apps de IA?
  • Proteger API keys: Nunca exponer credenciales en cliente
  • Controlar costes: Cada request cuesta dinero, necesitas saber quién gasta
  • Personalización: Guardar históricos, preferencias, contexto por usuario
  • RGPD/CCPA: Cumplimiento normativo
Auth.js (NextAuth)
Qué proporciona:
  • OAuth2 con múltiples proveedores (Google, GitHub, Discord...)
  • Email/Contraseña
  • Sesiones seguras (cookies httpOnly)
  • Middleware para proteger rutas
  • Callbacks personalizables
Flujo OAuth Típico
  1. Usuario selecciona proveedor (ej: Google)
  2. Redirección a Google (autorizar acceso)
  3. Google devuelve token de autorización
  4. Auth.js crea sesión segura (cookie httpOnly)
  5. Usuario puede acceder a rutas protegidas
Sessions vs JWT
Sessions JWT
✅ Más seguro 📦 Stateless
✅ Fácil revocar ⚡ Más escalable
Usa en: Next.js Usa en: APIs REST
Instalación Rápida
npm install next-auth

Luego: crear app/api/auth/[...nextauth]/route.ts y configurar proveedores

⏱️ Rate Limiting: Proteger tu Presupuesto ▼
¿Qué es y por qué CRÍTICO?
Definición: Controla cuántas requests puede hacer un usuario en un período (ej: 100/minuto)

¿Por qué es crítico?
  • Cada request a OpenAI, Anthropic, etc. cuesta dinero
  • Sin límites: atacante hace 10,000 requests = $10,000 en 1 minuto
  • DoS attack: hacer el servicio inusable para otros
  • Obligatorio en producción
Estrategias de Rate Limiting
  • Por Usuario: Cada account tiene cuota (ej: 1000/día). ✅ Justo | ⚠️ Requiere DB
  • Por IP: Limitar por dirección origen. ✅ Rápido | ⚠️ Fácil de evadir
  • Por API Key: Cada cliente obtiene clave con límite. ✅ Flexible | ⚠️ Gestión de claves
  • Global: Límite total del servicio. ✅ Simple | ⚠️ Poco granular
Implementación Básica (En Memoria)
Map<userId, [timestamp1, timestamp2...]>

Filtrar timestamps fuera de la ventana → si count >= limit → bloquear

⚠️ Solo para desarrollo. Pierde datos si reinicia el servidor.

Upstash: Rate Limiting en Producción
  • Redis serverless en la nube
  • Plan gratuito: 10,000 comandos/día
  • Integración perfecta con Next.js + Vercel
  • Muy rápido (Redis es ultrarrápido)
npm install @upstash/ratelimit redis
HTTP Status Codes
  • 429 Too Many Requests — Rate limit excedido
  • 401 Unauthorized — No autenticado
  • 403 Forbidden — Permisos insuficientes
🛡️ Proteger una API Route (Full Stack) ▼
Pasos Clave
  1. Autenticación: const session = await getServerSession()
  2. Rate Limiting: await checkRateLimit(userId)
  3. Operación costosa: Llamar a OpenAI (en servidor, API key protegida)
  4. Devolver resultado al cliente
Puntos Críticos de Seguridad
  • ❌ NUNCA expongas API keys en cliente (JavaScript, HTML)
  • ✅ SIEMPRE usa variables .env.local (no en Git)
  • ✅ SIEMPRE verifica sesión antes de operación costosa
  • ✅ SIEMPRE limita requests antes de llamar a APIs caras
Middleware para Proteger Rutas
Crea middleware.ts en la raíz:

Define qué rutas necesitan autenticación: /api/chat/:path*, /dashboard/:path*, etc.

Auth.js automáticamente redirige a signin si no hay sesión válida

Ejemplo Mínimo de API Route Segura
app/api/chat/route.ts
  1. Verificar sesión con getServerSession()
  2. Comprobar rate limit del usuario
  3. Llamar a OpenAI con credenciales del servidor
  4. Devolver respuesta
🏗️ Arquitectura de Seguridad Completa ▼
Componentes
  • Cliente (Navegador): UI, login button, almacena solo cookie de sesión (httpOnly)
  • Auth Server: Auth.js maneja login, devuelve sesión segura
  • API Routes: Protegidas con getServerSession()
  • Rate Limit Cache: Redis/Upstash para tracking rápido
  • Secrets Server: OpenAI key, variables .env.local
Flujo Típico de una Request Segura
  1. Usuario hace request a /api/chat
  2. Middleware valida sesión (cookie)
  3. API route llama a getServerSession()
  4. Rate limit check en Upstash
  5. Llamada a OpenAI (desde servidor, key protegida)
  6. Devolver respuesta al cliente
Variables de Entorno Necesarias
.env.local (NO versionado):
  • NEXTAUTH_URL=http://localhost:3000
  • NEXTAUTH_SECRET=aleatorio-secreto
  • GOOGLE_CLIENT_ID=xxxxx
  • GOOGLE_CLIENT_SECRET=xxxxx
  • OPENAI_API_KEY=sk-xxxxx
  • UPSTASH_REDIS_REST_URL=xxxxx
  • UPSTASH_REDIS_REST_TOKEN=xxxxx
Checklist de Seguridad
  • ✓ Middleware protege rutas sensibles
  • ✓ API keys en servidor, NUNCA en cliente
  • ✓ Rate limiting antes de operaciones costosas
  • ✓ Sesiones con httpOnly (no accesibles desde JS)
  • ✓ NEXTAUTH_SECRET en producción (aleatorio, largo)
  • ✓ .env.local en .gitignore
  • ✓ Logging de requests para auditoría
⚠️ Errores Comunes a Evitar ▼
Error #1: API Key en Cliente
❌ const response = await fetch('openai', { headers: { 'Authorization': 'Bearer ' + process.env.OPENAI_API_KEY } })

✅ const response = await fetch('/api/chat', { method: 'POST', body: JSON.stringify({ message }) })

process.env en el cliente es accesible a cualquiera que abra DevTools.

Error #2: Sin Rate Limiting
❌ Aceptar requests sin límite

✅ Implementar rate limit ANTES de operación costosa

Costo: $10,000 en 1 minuto si alguien spam 10,000 requests.

Error #3: Saltar Validación de Sesión
❌ Asumir que si llegó a la API, el usuario está autenticado

✅ SIEMPRE llamar a getServerSession()

Alguien podría hacer requests directas a tu API sin pasar por el cliente.

Error #4: Rate Limit en Memoria
❌ Usar un Map en memoria en producción

✅ Usar Upstash o Redis persistente

Al reiniciar el servidor, se pierden todos los límites.

Error #5: NEXTAUTH_SECRET Débil
❌ NEXTAUTH_SECRET=abc123

✅ NEXTAUTH_SECRET= (generar con: openssl rand -base64 32)

Token débil = sesiones falsificables.
